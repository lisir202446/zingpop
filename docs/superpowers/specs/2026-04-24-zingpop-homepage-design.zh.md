# Zingpop 首页改版设计

日期：2026-04-24  
范围：仅限 `packages/console/app/src/routes/index.tsx` 及首页直接相关的内容与素材  
状态：方案已在对话中确认，等待你审核这份中文稿

## 目标

把 `/` 首页改成一个完整、正式的网站首页，整体质感继续保持现有 `opencode` 官网的风格，但产品定位要明显转向“不会写代码的普通用户”，而不是企业采购方或资深开发者。

用户在一屏到一屏半的阅读里，应该马上感受到三件事：

1. Zingpop 是给不懂代码的普通人用的。
2. 首页最重要的动作是“马上开始”，也就是注册或登录。
3. Zingpop 之所以更容易上手，是因为很多关键能力已经内置好了，用户不用从空白开始硬摸索。

## 问题总结

当前首页不能作为最终版 Zingpop 官网首页，原因有这些：

- 它现在更像一个极简入口页，而不是正式官网首页。
- 主页主文案已经出现乱码。
- 首页主体内容绕开了现有 i18n 体系。
- 相比之前 `opencode` 首页的完整结构，现在页面缺少深度、可信度和官网应有的完整感。
- 当前表达过于抽象，对非技术用户来说，看不出“我到底能拿它做什么”。

## 产品定位

首页应该把 Zingpop 表达成这样一种产品：

它帮助不会写代码的用户，也能一步步做出看起来像样、能展示、能继续修改的产品结果，而不是要求用户先学会一大堆复杂流程。

品牌核心表达是：

`很多麻烦的准备工作，Zingpop 已经先帮你准备好了，所以你不用从空白开始，也不用一个人把整套流程全拼出来。`

首页可以直接提到这些内置能力名词，但只能作为“理念证据”，不能写成技术说明文档：

- `skill`
- `prompt`
- `context`
- `plugin`
- `mcp`

这些词出现时，必须同时配一层普通用户能看懂的解释，不能默认读者本来就懂这些词。

## 目标用户

主要用户：

- 个人用户
- 有想法，但几乎不会写代码的人
- 想更快做出看得见结果的人，比如页面、原型、小工具、创意展示

次要用户：

- 对产品感觉和页面质感有判断力，但不希望看到开发者味太重首页的人

明确不是这版首页主要面向的人：

- 企业采购方
- 走销售流程或商务流程的用户
- 想在首页看底层技术架构细节的人

## 非目标

这次首页改版不做这些事：

- 不把首页做成技术文档页
- 不以企业服务为主叙事
- 不详细解释 `skill`、`mcp`、`plugin` 的实现机制
- 不新增后端产品能力
- 不把整站视觉语言改离 `opencode` 现有风格
- 如果现有 `opencode` 官网结构可以复用，就不从零重做一套新首页

## 视觉方向

首页应该继续保留 `opencode` 官网那种克制、正式、像编辑型产品网站的感觉。

默认实现原则是：

`尽量复用现有 opencode 首页的设计语言，优先替换内容和素材。`

应该保留：

- sticky header
- 带边框的中间内容容器
- 长页分段结构
- 安静、清楚、高对比的排版
- 少量但有意义的动效
- FAQ、邮箱订阅、Footer、Legal 这套完整收尾结构

应该避免：

- 吵闹的创业站渐变风格
- 过多 badge 和增长黑客式 UI
- 太卡通、太跳脱的视觉
- 没有叙事节奏、只是不断堆功能卡片

整体感觉应该还是 `opencode` 的同一家族，只是内容上更温和、更面向新手。

这也意味着，这次改版应该优先“换内容”，而不是“重做视觉系统”。

具体原则：

- 能复用现有 section 节奏，就不重排一整套新布局
- 能复用现有共享组件，就先复用，不轻易新增
- 能复用现有 CSS 模式，就不另起一套新的视觉系统
- 优先替换文案、插图、案例截图，而不是重做基础布局骨架
- 只有在现有视觉明显挡住“面向新手”的定位时，才做必要的最小样式调整

## 首页结构

### 1. Header

继续复用现有官网头部结构，让用户一打开就知道这是一个正式网站，而不是临时入口页。

要求：

- 顶部固定
- 主 CTA 在头部里可见
- 导航尽量短、清楚、好理解

导航意图：

- 首页
- 文档或新手指南
- 登录
- 主按钮：`立即开始` 或同类、对新手更友好的文案

如果现有共享 Header 的局部文案还是太偏开发者，可以只调整首页里的呈现方式和 CTA 语气，不去改动无关页面。

### 2. Hero 首屏

作用：

- 清楚表达“这是给新手的”
- 让用户觉得点进去是安全的、可开始的
- 明确最重要的转化动作

内容方向：

- 主标题必须是结果导向的人话，不是技术词
- 副标题解释：Zingpop 帮你更容易做出像样的结果，不需要你先懂复杂操作
- 主按钮：`立即开始 / 注册登录`
- 次按钮：`看它能做什么`

语气要求：

- 有信心
- 让人安心
- 足够具体
- 不技术化

首屏绝不能再像开发者工具安装页。

### 3. 结果导向案例区

作用：

- 直接让新手看到“我能做出什么”
- 用结果取代抽象的能力描述

这一块紧接在 Hero 后面。

必须包含 4 类案例：

- 小游戏
- 个人主页
- 小工具
- 艺术展示

呈现方式：

- 4 张卡片或 4 个案例块
- 每张卡都配更像真实产品界面或截图的视觉，而不是抽象概念图
- 每张卡都要有一句普通用户能读懂的结果描述

配图方向：

- 走“像真实产品截图/界面展示”的路线
- 实现时用 OpenAI `gpt-image-2` 生成素材
- 页面本身不对用户强调模型名

文案方向：

- 重点写用户最后能得到什么
- 例如：从一个想法开始，先做出一个看得见、能分享、还能继续修改的结果

### 4. 理念区

作用：

- 解释为什么新手也能在 Zingpop 里做出来东西
- 说明产品的核心差异，但不把页面写成技术说明

核心表达：

`Zingpop 会先把很多难的部分准备好，所以你不用从零拼流程。`

这一块可以出现：

- `skill`
- `prompt`
- `context`
- `plugin`
- `mcp`

但每个词都必须翻译成用户收益：

- `skill`：常见任务的做法已经先准备好
- `prompt`：不用自己从零想怎么提问
- `context`：前面已经说过的内容可以接着做
- `plugin`：以后能力可以继续扩展，不用换工具
- `mcp`：以后接更多工具和资料时，路径也是预留好的

这一块应该让用户觉得“放心”，而不是让用户觉得“听不懂”。

### 5. 能力列表区

作用：

- 保留旧 `opencode` 首页那种“能力很扎实”的感觉
- 但表达方式改成用户收益导向

这一块应该有旧官网那种清单式可信感，但文案是给普通用户看的。

可以列的方向：

- 可以接着上一步继续做，不用每次重来
- 能处理长内容和多轮修改
- 能帮助做页面、原型、小工具和创意展示
- 很多能力会一起配合，不用你自己拼流程
- 后续会越来越多现成任务可以直接开始
- 工作台会越来越接近“点一下就能开始”的方式

这一块要看起来有分量，但不能读起来太累。

### 6. 打开就能开始的工作台预告

作用：

- 提前告诉用户未来工作台会更容易上手
- 让产品看起来不像只有一个空白输入框

核心表达：

工作台会逐步加入现成可点的开始入口，里面会预置好提示词和 skill，用户可以直接点击任务开始，而不是对着空白框发呆。

可展示的任务方向：

- 做一个主页
- 做一个小游戏原型
- 生成一个实用小工具
- 做一个展示页面
- 改写和优化已有内容

这一块可以写成产品方向，但语气要诚实：

- 可以表达“这是正在往这个方向做”
- 不能写成“这些体验已经全部完整上线”

### 7. 新手三步开始

作用：

- 降低心理门槛
- 让用户觉得上手非常简单

三步流程：

1. 注册登录
2. 选择一个你想做的任务
3. 边看结果边继续修改

这块应该非常好懂，用户一眼就能读完。

### 8. FAQ

作用：

- 在用户离开前解决最常见的新手顾虑

问题方向建议：

- 我不会写代码也能用吗？
- 我不知道怎么开头怎么办？
- 能不能直接选现成任务开始？
- 做出来之后还能继续改吗？
- 以后会不会有更多可直接使用的任务？

尽量复用现有 FAQ 组件和首页收尾结构。

### 9. Email Signup、Footer、Legal

继续复用现有官网的尾部结构，把首页完整收住：

- Email signup
- Footer
- Legal

这是必要的，否则首页还是会显得像做了一半。

## 文案原则

首页所有文案都遵循这些原则：

- 先讲结果，再讲系统
- 先讲人话，再讲产品名词
- 技术词只有在能带出用户收益时才出现
- 不要企业化措辞
- 不要 API-first 或 model-first 的表达
- 让用户一直感受到：更安心、更有进度、更能看见结果

整页应该反复回答三个问题：

- 我能做出什么？
- 为什么这里更容易？
- 我现在该点哪里开始？

## i18n 要求

首页必须重新回到项目现有的 i18n 体系中。

要求：

- 不能继续把首页主体内容塞在一个硬编码静态对象里
- 不能再出现“导航是翻译的，主体文案是硬写的”这种混合状态
- 需要新增或复用 locale key，保证首页在支持语言里结构一致
- 必须彻底去掉当前乱码内容

如果一次性补全所有语言成本太高，后续实现计划里也必须保住 i18n 架构，不能再走静态硬编码绕过去。

## 组件复用策略

优先复用现有营销站的结构和组件：

- `Header`
- `Faq`
- `EmailSignup`
- `Footer`
- `Legal`
- 旧 `opencode` 首页已有的 section 样式模式

但不要为了复用而强行保留旧首页里那些太开发者导向的内容块。优先复用布局和气质，其次才是复用旧内容结构。

实现优先级应该是：

1. 先复用现有 `opencode` 首页的布局和区块框架
2. 再替换成符合 Zingpop 新手定位的文案
3. 再替换成符合 Zingpop 案例方向的素材
4. 只有在复用明显走不通时，才新增结构或样式代码

## 素材策略

案例区的视觉素材应该作为本地素材加入仓库，而不是在浏览器里实时生成。

计划：

- 用 OpenAI `gpt-image-2` 生成截图风格案例图
- 选定版本后存为本地素材文件
- 首页卡片直接使用这些本地素材

这样既能保留更高质量视觉，也不会让线上运行依赖实时生成。

## 验收标准

这次改版成功的标准是：

1. `/` 看起来像一个完整、正式的官网首页，而不是极简入口页。
2. 用户第一眼看到的最重要动作是 `立即开始 / 注册登录`。
3. Hero 后的第一大块内容，是“用户能做出什么”的案例展示。
4. 页面明确面向新手，而不是企业采购或技术评估。
5. 理念区清楚表达：很多关键基础能力已经内置好，并且能用普通用户能懂的方式提到 `skill / prompt / context / plugin / mcp`。
6. 整体质感继续保持旧 `opencode` 首页的完成度和正式感。
7. 乱码内容被彻底清除。
8. 首页主体内容重新回到 i18n 体系。
9. 页面尾部完整包含 FAQ、邮箱订阅、Footer、Legal。
10. 最终实现大体上仍然建立在 `opencode` 首页原有结构和设计模式上，改动重点放在内容和素材，而不是整套视觉重做。

## 验证计划

后续实现时，应按这些方式验证：

1. 在桌面和移动宽度下做首页视觉检查
2. 代码层面确认首页主体内容不再依赖当前那个静态 `homePage` 对象
3. 在 `packages/console/app` 下运行包级 typecheck
4. 如果本地构建环境允许，再补一次生产构建验证

## 实现范围边界

这份设计只聚焦首页及其直接依赖的文案和素材。

不包括：

- 工作台本身的功能重做
- 真正把“点一下就开始”的任务系统完整上线
- 重写无关路由
- 把整站导航系统全部改掉

后续实现应该只做一次高质量首页改版，尽量避免无关改动。
